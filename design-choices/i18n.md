# Design choice — Internationalisation & RTL

**Decision:** translate with **`@solid-primitives/i18n@^2`**, format with native **`Intl`**, and handle RTL
(Arabic) with a **`dir`/`lang` attribute on `<html>`** + **Tailwind logical utilities** — no i18n library
beyond the tiny primitive, no formatting library. (Spec: `../specs/i18n/SPEC.md`.)

**Why:**
- **`@solid-primitives/i18n` is a handful of pure functions** (`flatten`, `translator`, `resolveTemplate`) over
  a reactive dictionary memo — tiny, tree-shakeable, reactive, type-safe (keys typed from the base dictionary).
  Locale switches update the UI with no extra machinery. Peer `solid-js ^1.6` → fine on 1.9 (avoid the
  `3.0.0-next` Solid-2.0 prerelease).
- **Native `Intl`** (`NumberFormat`/`DateTimeFormat`/`Collator`/`PluralRules`) gives locale-correct numbers/
  dates at **zero bundle cost**.
- **RTL is a layout concern, not the i18n lib's** — set `dir="rtl"` for RTL locales and let Tailwind v4's
  `rtl:` variants + logical properties mirror the layout automatically (composes with theming).

## Costs / trade-offs
- **You own the plumbing** — a small `I18nProvider` (locale signal + persistence + `<html lang/dir>` effect),
  the `Intl` helper wrappers, and locale detection. The primitive gives you `t`, not a framework.
- **Component discipline** — every user-facing string must go through `t()`, and every component must use
  **logical CSS** (no physical `left/right`) or RTL won't mirror. This is the same app-wide sweep as theming's
  token conversion — done once, together (`../specs/conventions.md`).
- **Dictionary maintenance** — non-English dicts are type-checked against the base (missing keys fail the
  build), which is safe but means every key must be translated (or explicitly fall back). Lazy-load non-default
  locales to keep the initial bundle small.
- **RTL edge cases** — directional icons must mirror (`rtl:-scale-x-100`), left-anchored panels (mobile drawer)
  must use inline-start; Arabic-Indic vs Latin digits is an `Intl` locale choice.

## Alternatives rejected
- **i18next (+ Solid binding)** — heavier and more than we need.
- **A formatting lib (date-fns/numbro)** — `Intl` is built-in and locale-correct; no reason to ship one.

## Net
Smallest viable stack: a tiny reactive translator + built-in `Intl` + CSS-driven RTL. Cost is app-wide `t()` +
logical-CSS discipline (shared with the theming sweep). Parallel-safe with theming — see
`../specs/conventions.md`.
